  <behavior>
    createLLMAdapter(provider, apiKey, modelId):
    - Uses createOpenAI({ apiKey }) from '@ai-sdk/openai' when provider='openai' — NEVER reads OPENAI_API_KEY from env
    - Uses createAnthropic({ apiKey }) from '@ai-sdk/anthropic' when provider='anthropic' — NEVER reads ANTHROPIC_API_KEY from env
    - Unknown provider throws Error('Unknown provider: ${provider}. Supported: openai, anthropic')
    - Returns an LLMAdapter object implementing streamText and generateStructured

    LLMAdapter.streamText(messages, onToken):
    - Calls streamText() from 'ai' package with the model and messages
    - Iterates result.textStream async iterable
    - Calls onToken(delta) for each delta
    - Returns the full accumulated text string
    - On network error: wraps in user-friendly message and rethrows

    LLMAdapter.generateStructured(messages, schema):
    - Primary: calls generateText() from 'ai' with Output.object({ schema }) — AI SDK v6 API
    - Returns output field from result
    - Fallback (on primary failure): calls generateText() without structured output to get raw text
      - Strips common fenced code block wrappers (```json...```) using regex before jsonrepair
      - Calls jsonrepair() on extracted text
      - Parses JSON.parse(repaired)
      - Validates with schema.parse() — if this also throws, rethrows with message 'LLM returned unparseable response after fallback attempt'

    Message type: { role: 'user' | 'assistant' | 'system'; content: string }
    All messages passed to AI SDK must be typed as CoreMessage from 'ai' package

    CRITICAL: Never call provider constructors without explicit apiKey (research pitfall 6).
    CRITICAL: Use AI SDK v6 Output.object() API, not standalone generateObject() (research state-of-the-art table).
  </behavior>

  <name>Task 2: Streaming display renderer with gutter indicator and separator</name>
  <files>src/display/stream.ts</files>
  <behavior>
    streamResponse(adapter, messages, options?):
    - Default options: gutterChar='│', separatorChar='─', separatorWidth=60
    - Writes dim gutter prefix to stdout BEFORE first token: process.stdout.write(chalk.dim(gutterChar + ' '))
    - Calls adapter.streamText(messages, onToken) where onToken writes delta directly to stdout via process.stdout.write(delta)
    - After streaming completes (streamText returns), writes newline then separator line: '\n' + chalk.dim(separatorChar.repeat(separatorWidth)) + '\n'
    - Returns the full text from adapter.streamText

    StreamOptions interface:
    - gutterChar?: string
    - separatorChar?: string
    - separatorWidth?: number

    Interruption handling (per user decision):
    - Wraps streamText call in try/catch
    - On error (including SIGINT-like aborts): prints chalk.yellow('\n[Stream interrupted. Use `req chat <id>` to retry the last round.]')
    - Rethrows the error so caller can handle

    CRITICAL — spinner conflict avoidance (per research pitfall 3):
    - streamResponse does NOT manage any spinner
    - The caller is responsible for stopping any spinner BEFORE calling streamResponse
    - streamResponse only writes the gutter prefix + tokens + separator — nothing else on stdout

    Chinese character display:
    - process.stdout.write handles UTF-16 strings from LLM natively — no special encoding conversion needed
    - If on Windows and process.platform === 'win32' and process.stdout.encoding !== 'utf8': print a one-time warning via stderr suggesting Windows Terminal
  </behavior>
